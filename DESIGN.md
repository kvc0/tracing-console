# tracing-cache

`tracing-cache` is a `tracing::Subscriber` that holds spans in memory and exposes them for inspection. It is designed to back a console or UI that needs to browse, filter, and page through live span data.

## Affordances

- **Hierarchical enable/disable**: If a span is disabled, all spans and events created within it are also disabled. This is enforced in `Subscriber::enabled()` using a thread-local span stack.
- **Predicate-based filtering**: A user-supplied `EnabledPredicate` trait object controls which spans are recorded. Applied in `enabled()` after the hierarchy check, to optionally filter out more spans.
- **Flexible-cost filtering**: the `EnabledPredicate::new_span_enabled` allows users to decide span enable status based on last-moment dynamic span attributes.
- **Event capture**: `tracing::event!` calls occurring within a recorded span are attached to that span's record.
- **Bounded memory**: Holds at most `capacity` spans. When full, the oldest span (lowest id) is evicted.
- **Paginated access**: Spans are accessible by id (`get_span`) or in pages (`page(after_id, limit)`).

## Core types

### `EnabledPredicate`

```rust
pub enum Interest {
    Never,
    Sometimes,
    Always,
}

pub trait EnabledPredicate: Send + Sync + 'static {
    /// as in tracing::Subscriber, this is an efficient way to skip all processing of callsites beyond a particular level.
    fn max_level_hint(&self) -> Option<LevelFilter>;
    /// Inspect whatever metadata to decide if this callsite should ever be checked again. It is called for both spans and events.
    /// Never -> it will never be checked again, it is a disabled site for both spans and events.
    /// Sometimes -> callsite enabled and new_span_enabled will be consulted for spans. For events, the callsite is unconditionally enabled. Note, however, that Spans are not recorded on a current span that is disabled.
    /// Always -> only consult new_span_enabled for this site for spans. For events, they will just be unconditionally enabled.
    fn callsite_enabled(&self, metadata: &'static Metadata<'static>) -> Interest;
    fn enabled(&self, metadata: &Metadata<'_>) -> bool;
    fn new_span_enabled(&self, span: &span::Attributes<'_>) -> bool;
}
```

Implement this to filter spans by level, target, or any other metadata. A default `LevelPredicate` implementation is provided.

### `SpanRecord`

```rust
pub struct SpanRecord {
    pub id: u64,
    pub parent_id: Option<u64>,
    pub metadata: &'static Metadata<'static>,
    pub fields: HashMap<&'static str, String>,
    pub events: Vec<EventRecord>,
    /// set at new_span()
    pub opened_at: Instant,
    /// set at try_close()
    pub closed_at: Option<Instant>,
}
```

### `EventRecord`

```rust
pub struct EventRecord {
    pub metadata: &'static Metadata<'static>,
    pub fields: HashMap<&'static str, String>,
    pub recorded_at: Instant,
}
```

### `SpanCache`

```rust
pub struct SpanCache<P: EnabledPredicate = LevelPredicate>;

impl SpanCache<LevelPredicate> {
    pub fn new(capacity: usize) -> Self;
}

impl<P: EnabledPredicate> SpanCache<P> {
    pub fn with_predicate(capacity: usize, predicate: P) -> Self;

    // Accessor API
    pub fn get_span(&self, id: u64) -> Option<SpanRecord>;

    /// after_id: 0 -> start from the lowest id
    /// after_id: N -> start after the id specified (probably sourced from a previous call to page)
    pub fn page(&self, after_id: u64, limit: usize) -> Vec<SpanRecord>;
}
```

## Internal design

### Storage

```
SpanCache {
    state: Mutex<State>,       // guards spans
    next_id: AtomicU64,        // outside the lock; relaxed increments
    predicate: P,
    capacity: usize,
}

State {
    spans: BTreeMap<u64, SpanRecord>,  // ordered by id for eviction and pagination
}
```

The `BTreeMap` gives O(log n) insert/lookup, O(1) pop-first eviction, and range queries for pagination without a secondary index.

### Span id generation

Ids are generated by a per-`SpanCache` `AtomicU64` starting at 10, incremented with relaxed ordering. The resulting `u64` is wrapped in `tracing::span::Id` via `Id::from_u64`. Because ids are monotonically increasing, the smallest id in the `BTreeMap` is always the oldest span.

Span ID `1` is special: It means `DISABLED`. It is set whenever a new disabled span is created.

When `DISABLED` is enter()'d, it is pushed onto the span stack. Same with exit(DISABLED). record, try_close, etc. are no-ops with `DISABLED`. It is put on the stack so that when this span's DISABLED scope exits and the new stack tail is not DISABLED, other spans might be eligible for being enabled.

The parent id comes from the new_span call. It is computed based on the passed-in Attributes's Parent.
* If a span's attributes `is_contextual()`, and if there is not a current active span or the current span is disabled, then this new span is disabled. If it `is_contextual()` and there is a current span, then the parent_id is the current span id.
* If a span `is_root`, and if there is not any current span, then create the new root span. If there is already an existing span stack, then `log::warn` and this new span is disabled. This subscriber tracks only 1 span stack at a time: The warn is logged to help developers straighten out their stacks and have tidy span hygiene. This is meant for API developers where an API call is a root span.
* If a span's attributes has an explicit parent id, then that is the new span's parent_id. If the parent span is `DISABLED` then so is this span.

When `enabled()` is called:
1. Check the tail of the thread-local span stack. If that id is `DISABLED` → disabled.
2. Ask the `EnabledPredicate::enabled`. If false → disabled.
3. Otherwise → enabled.

This means spans and events created while inside a disabled span are silently dropped. That is intentional.

When `new_span()` is called:
1. Check the tail of the thread-local span stack, or the explicit parent id instead if set. If that id is `DISABLED` → disabled. With empty stack and no explicit parent, continue to contextual/root...
2. If the new span is_contextual and the span stack is empty -> disabled.
3. If the new span is_root and the span stack has content -> disabled.
4. Ask the `EnabledPredicate::new_span_enabled`. If false → disabled.
5. Otherwise → enabled.

### `enabled()` and the span stack

A thread-local `Vec<u64>` tracks the stack of active span ids on each thread. `Subscriber::enter` pushes; `Subscriber::exit` pops. The thread-local span stack is wrapped in a `RefCell` and accessed without `unsafe` and without locking, via `with()` closures.

It is possible to have a span stack like `[123, 124, 1, 1, 1]`. That would mean that the root span 123 has a subspan 124 that is currently entered. Under 124, there is another span that was disabled for whatever reason, e.g., level. Beneath that disabled span, there is another subspan and yet another beneath it. These final 2 spans are short-circuited to disabled because their parent is disabled, so we don't need to consider their attributes to decide if they are enabled.
If we pop to `[123, 124, 1, 1]` then `[123, 124, 1]`, then to `[123, 124]`, then a new span is created, we see the current span is 124, and we should evaluate enabled() status. If it is enabled then we'd be able to get a stack like `[123, 124, 133]` or somesuch.

### Eviction

When inserting a new span:

```
while capacity <= spans.len() && spans.peek().is_closed() {
    spans.pop_first()
}
```

But in order to make sure active spans aren't dropped, first check if the span to be evicted is closed. If it's not closed, then let the new span be unconditionally DISABLED. Also log a warning here - it means
that span rate is too high for your configured span buffer size. The cache will eventually become smaller when the span drops. However, we should make it clear in documentation that this product is for API spans - spans that complete in a finite amount of time.

### Field capture

A local `FieldVisitor` implementing `tracing::field::Visit` converts all field types to `String` via their `Display` impl. The same visitor is used for both span fields (in `Subscriber::record`) and event fields (in `Subscriber::event`).

### Event routing

When `Subscriber::event` fires, the parent span id is resolved in order:

1. `event.parent()` (explicit parent set at the callsite)
2. Top of the thread-local span stack (current span)

If a parent is found and is not disabled, the `EventRecord` is appended to that span's `events` vec.
If the parent is not found, or it is disabled, or the event stack is empty, the event is dropped with a log::debug.
